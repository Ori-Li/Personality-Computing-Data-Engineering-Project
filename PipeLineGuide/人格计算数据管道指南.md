
# Personality Computing 数据生产 Pipeline v1.2

## 一、总体目标

本 Pipeline 用于人格计算项目的数据生产，目标包括：

- 人物实体生成
- 作品实体生成
- 人物与作品关系建立
- CPR 心理向量标注
- 图片抓取与存储
- 全流程支持断点恢复、去重、增量生成

---

# 二、总体流程

```text
人物名单
    │
    ▼
Person Collection Agent
    │
    ▼
人物 JSON（临时）
    │
    ▼
Person Entity Resolution（数据库查重）
    │
    ▼
保存 Person
    │
    ▼
遍历人物代表作品
    │
    ▼
Work Entity Resolution（数据库查重）
    │
    ├── 已存在 → 获取 workId
    │
    └── 不存在
            │
            ▼
      Work Collection Agent
            │
            ▼
      保存 Work
            │
            ▼
建立 Person_Work 关系
            │
            ▼
删除临时 JSON
            │
            ▼
CPR Annotation Pipeline
            │
            ▼
更新 Person / Work 心理向量
            │
            ▼
Image Crawling Pipeline
            │
            ▼
OSS 对象存储
            │
            ▼
更新图片 URL
```

---

# 三、Pipeline 1：人物生成

输入：

- 人物名称

输出临时 Person JSON：

- 基础信息
- 简介
- 代表作品列表

> 此阶段**不生成心理向量**。

---

# 四、Pipeline 2：人物实体消歧（Entity Resolution）

推荐组合键：

```
name
+
primaryField
+
country
```

如果数据库已存在：

- 获取 personId

否则：

- INSERT Person
- 返回 personId

---

# 五、Pipeline 3：作品处理

遍历该人物所有作品。

## 1. Normalize

生成：

- workHash

## 2. 数据库查重

推荐顺序：

```
workHash
↓

title
↓

title + type
↓

title + type + country
↓

title + type + country + year
↓

primaryCreator
```

如果存在：

- 获取 workId

否则：

- 调用 Work Collection Agent
- 生成完整 Work JSON
- INSERT Work
- 获取 workId

---

# 六、Pipeline 4：建立人物-作品关系

关系表：

```
person_work
```

字段：

- personId
- workId
- relationType

始终通过 **ID** 建立关系，不通过名称建立关系。

---

# 七、Pipeline 5：临时 JSON 生命周期

Person JSON：

```
生成
↓

保存 Person

↓

处理所有作品

↓

建立关系

↓

删除 JSON
```

Work JSON：

```
生成

↓

保存 Work

↓

删除 JSON
```

JSON 仅作为 Agent 间的数据交换格式，不长期保存。

---

# 八、Pipeline 6：CPR 心理向量

Collection 完成后：

查询：

```
status = WAITING
```

生成：

- semantic
- experience_vector
- psychology_vector
- media_vector
- personality_affinity
- supporting_evidence

随后 UPDATE 数据库。

Collection 与 CPR 完全解耦。

---

# 九、Pipeline 7：图片流程

独立异步执行：

```
Entity
↓

搜索图片

↓

下载图片

↓

上传 OSS

↓

更新 image_url
```

图片抓取不依赖 LLM。

---

# 十、数据库设计

## Person

- person_id
- name
- primary_field
- country
- birth_year
- entity_key

## Work

- work_id
- title
- original_title
- type
- subcategory
- country
- year
- primary_creator
- work_hash
- entity_key

## Person_Work

- person_id
- work_id
- relation_type

---

# 十一、实体消歧规则

## Person

推荐唯一组合：

```
name
+
primaryField
+
country
```

必要时增加：

- birthYear
- activePeriod

## Work

推荐唯一组合：

```
workHash
+
type
+
country
+
year
```

必要时增加：

- primaryCreator

---

# 十二、优势

- JSON 不会越来越大
- 支持断点续跑
- 支持增量生成
- 作品自动去重
- 人物与作品采用多对多关系
- CPR 与 Collection 完全独立
- 图片流程独立
- 可扩展至几十万实体
- 后续可接入 Redis、Bloom Filter、向量数据库
